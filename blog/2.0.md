# Audit Log 2.0: a durable, searchable, tamper-evident audit trail for Laravel

*Announcing `boldlygrow/audit-log` 2.0 ŌĆö July 2026*

`boldlygrow/audit-log` gives Laravel apps a single, consistent way to record audit events ŌĆö a predefined set of context keys, a formatted array returned from every call, and clean, greppable log output.

Version 2.0 is the biggest release yet. Beyond the rebrand from `provisionesta/audit`, it delivers the durable database layer earlier versions promised, and turns the package into a proper audit *store* ŌĆö one you can query, that encrypts sensitive fields, and that can be locked down to be tamper-evident for compliance. Every new capability is opt-in with backward-compatible defaults, so existing `AuditLog::create()` call sites keep working exactly as before.

Here are the highlights.

## 1. A real, queryable database trail

System logs get rotated, sampled, and shipped off to third parties. Audit events often need to stick around ŌĆö perpetually, and searchable with SQL. 2.0 adds optional database persistence: pass `database: true` and the event is written to an `audit_logs` table in addition to the system log.

```php
AuditLog::create(
    event_type: 'okta.user.updated.success',
    level: 'info',
    message: 'User profile updated',
    method: __METHOD__,
    record_id: $user->id,
    record_model: \App\Models\User::class,
    database: true,
);
```

The package ships a working Eloquent model and a publishable migration, so it works out of the box after `vendor:publish` + `migrate`. Persistence is enabled by default; if the table hasn't been migrated yet, the event still lands in the system log and you get an actionable warning instead of an exception.

## 2. Immutable, tamper-evident records

An audit trail you can quietly edit isn't much of an audit trail. 2.0 makes persisted records immutable by default: updates and permanent deletes (`forceDelete`) are blocked and throw an `ImmutableRecordException`.

```php
$entry->update(['message' => '...']); // throws ImmutableRecordException
$entry->forceDelete();                // throws ImmutableRecordException
```

Soft deletes are always allowed ŌĆö and every soft delete, restore, and (where you've explicitly permitted it) update is itself recorded as a new audit entry noting what happened and which actor did it. The mutation and the entry that records it are wrapped in a single transaction, so a record is never changed without its log. You can attach a reason, too:

```php
$entry->withJustification('Redacted PII per DSAR #4567')->delete();
```

Loosen the policy per-application with `config('audit-log.immutable.update')` and `.destroy` (both default `true`).

## 3. Encryption at rest for sensitive fields

Audit records tend to accumulate PII and before/after values. The shipped model now encrypts the fields most likely to hold sensitive data ŌĆö `actor_email`, `actor_name`, `actor_username`, `attribute_value_old`, `attribute_value_new`, `parent_reference_value`, `record_reference_value`, and the `metadata` array ŌĆö using Laravel's `encrypted` casts, stored in `TEXT` columns. The model decrypts transparently on read.

## 4. Search encrypted fields without giving up encryption

Encrypting a column normally makes it unsearchable ŌĆö Laravel's encryption is non-deterministic, so `where('actor_email', ...)` can never match. 2.0 restores lookups with a set of query scopes that decrypt in PHP, build a short-lived cached index, and constrain the query to the matching rows.

```php
// Case-insensitive exact, partial, starts-with, ends-with on encrypted strings
AuditLog::whereEncryptedStringExact('actor_email', 'jsmith@acme.com')->get();
AuditLog::whereEncryptedStringPartial('actor_email', 'acme.com')->get();

// Search inside the encrypted metadata array
AuditLog::whereEncryptedArraySearch('metadata', 'acme')->get();
AuditLog::whereEncryptedArrayExact('metadata', 'approved_by', 'John Smith')->get();
```

Every scope returns a query builder, so it chains with normal `where`, `orderBy`, and `paginate`.

## 5. Pass a model, get a type

Hand-writing `record_type: 'okta_user'` next to a `record_id` is repetitive and drifts when models move. Now you pass the class and the snake_case type is derived for you:

```php
record_model: \App\Models\Okta\User::class, // record_type => "okta_user"
```

The same applies to `parent_model`, `related_model`, `subject_model`, and `tenant_model`. The legacy `*_type` string parameters still work; the model value wins when both are supplied.

## 6. Richer actor and relationship context

Events rarely involve just one authenticated user and one record. 2.0 adds `related_*` and `subject_*` relationship fields, and captures where a request came from with `actor_source` (`system`, `api`, `web`, or `cli`). Actor identity is auto-populated from the authenticated user ŌĆö and if your user model uses different column names, map them once in `config('audit-log.actor.attributes')` (a single attribute or an ordered fallback list) instead of forking the package.

## 7. Built for compliance conversations

Audit logging is usually adopted to satisfy a control. The new [`COMPLIANCE.md`](../COMPLIANCE.md) maps the package's capabilities to SOC 1/2, ISO 27001, NIST SP 800-53/63/171, CISA, and CIS ŌĆö with an explicit shared-responsibility model that's honest about what a logging library does and does not provide (integrity protection, retention enforcement, access control, and review remain your responsibility).

## More improvements

Higher-impact changes that don't need a section of their own:

- **Polymorphic Eloquent relationships** ŌĆö `actor()`, `parent()`, `record()`, `related()`, `subject()` resolve straight off the stored model class.
- **Custom columns without forking** ŌĆö whitelist keys in `config('audit-log.database.custom_fields')` and they flatten out of `metadata` into their own indexed columns.
- **Extend, don't fork** ŌĆö point `config('audit-log.database.model')` at your own model that extends the base to add relationships, casts, and scopes; package upgrades flow through.
- **Configurable proxy/CDN IP header** ŌĆö capture the real client IP behind Cloudflare or a load balancer.
- **Configurable table name** and a `withJustification()` helper for recording *why* a mutation happened.
- **Cleaner output** ŌĆö ISO 8601 Zulu timestamps and a class-only log message prefix (`ApiClient Success`), with the fully-qualified method still in the context.
- **`database` replaces `transaction`** ŌĆö clearer name; `transaction` stays as a deprecated alias.
- **Persistence never breaks your code** ŌĆö a misconfiguration logs a warning and continues.
- **Set-and-forget config** ŌĆö plain booleans in `config/audit-log.php` (the old `AUDIT_*` env vars are gone); database persistence is on by default.
- **Date-stamped migration publishing** and separate `audit-log`, `audit-log-config`, and `audit-log-migrations` publish tags.
- **Quality** ŌĆö a full Pest test suite and Larastan level-9 static analysis.

## Upgrading

2.0 raises the floor to **Laravel 11+ and PHP 8.2** (the shipped model relies on Laravel 11's `casts()` method ŌĆö on 8ŌĆō10 the casts were silently ignored, so the old range advertised support it didn't have), and renames a few things:

- Package: `provisionesta/audit` ŌåÆ `boldlygrow/audit-log` (kept as a `replace` alias).
- Class: `Log` ŌåÆ `AuditLog` (the old `Log` class remains as a deprecated subclass).
- Config: `config/audit.php` ŌåÆ `config/audit-log.php`.

Applications still on Laravel 8ŌĆō10 should stay on `provisionesta/audit:^1.3`. Full step-by-step notes, including the rationale for each change, are in the [2.0 release notes](../CHANGELOG.md#200---2026-07-20).
